# v0.8 Design Spec — Lightweight Sigma Rule Evaluator

> Status: Draft
> Author: d3vn0mi + Claude
> Date: 2026-04-17

---

## 1. Problem Statement

ArtiForge's detection system uses 13 hand-written Python lambda rules. These
work well but use a proprietary format. Sigma is the industry-standard,
SIEM-agnostic detection language that SOC teams use to write and share
detection rules. Supporting Sigma means:

- Lab authors write rules in the same format trainees will use professionally
- Rules are portable — convertible to KQL, SPL, or Lucene via `sigma-cli`
- Trainees can test their own Sigma rules against generated bundles

## 2. Solution

A lightweight, pure-Python Sigma YAML evaluator built into ArtiForge's
existing detector system. No pySigma dependency. Supports ~80% of Sigma
syntax — the subset that lab authors will actually use. Unsupported features
produce a warning and skip the rule, not an error.

## 3. Design Goals

- Zero external dependencies (beyond PyYAML, already a project dependency)
- Supplement built-in rules, not replace them
- Labs can ship Sigma rules in a `sigma/` subdirectory (auto-discovered)
- CLI: `--sigma-dir` for external rules, `--sigma-only` to skip built-in rules
- Clear output distinguishing built-in vs Sigma rule results

## 4. Supported Sigma Syntax

### 4.1 Logsource

```yaml
logsource:
    product: windows
    service: sysmon        # maps to channel "Sysmon"
    category: process_creation  # maps to channel + EID filter
```

**Logsource → channel/EID mapping:**

| product | service | category | Channel | EID filter |
|---------|---------|----------|---------|------------|
| windows | sysmon | process_creation | Sysmon | 1 |
| windows | sysmon | network_connection | Sysmon | 3 |
| windows | sysmon | file_event | Sysmon | 11 |
| windows | sysmon | registry_event | Sysmon | 12, 13, 14 |
| windows | sysmon | dns_query | Sysmon | 22 |
| windows | sysmon | process_access | Sysmon | 10 |
| windows | sysmon | image_load | Sysmon | 7 |
| windows | sysmon | pipe_created | Sysmon | 17, 18 |
| windows | sysmon | file_delete | Sysmon | 23, 26 |
| windows | sysmon | driver_load | Sysmon | 6 |
| windows | sysmon | create_remote_thread | Sysmon | 8 |
| windows | sysmon | process_tampering | Sysmon | 25 |
| windows | sysmon | clipboard_change | Sysmon | 24 |
| windows | security | — | Security | all |
| windows | system | — | System | all |
| windows | powershell | — | PowerShell | all |
| windows | — | process_creation | Security | 4688 |

When `service` is specified without `category`, match all EIDs on that channel.
When `category` is specified, filter to the mapped EIDs.

### 4.2 Detection Selections

```yaml
detection:
    selection:
        Image|endswith: '\cmd.exe'
        CommandLine|contains: 'whoami'
    condition: selection
```

**Field matching:** Each field in a selection is matched against `event_data[field_name]`.

**Value types:**
- Exact string: `'cmd.exe'` — case-insensitive match
- Wildcard: `'*payload*'` — `*` matches any substring
- List (OR): `['cmd.exe', 'powershell.exe']` — any value matches

**Modifiers** (applied to field name with `|`):
- `|contains` — value is a substring of the field
- `|startswith` — field starts with value
- `|endswith` — field ends with value
- `|all` — when combined with a list, ALL values must match (AND instead of OR)

**Within a selection, all fields are AND-ed.** A selection matches an event
only if every field in the selection matches.

### 4.3 Multiple Named Selections

```yaml
detection:
    selection_process:
        Image|endswith: '\cmd.exe'
    selection_network:
        DestinationPort: '443'
    condition: selection_process or selection_network
```

### 4.4 Filters (Exclusions)

```yaml
detection:
    selection:
        Image|endswith: '\svchost.exe'
    filter:
        ParentImage: 'C:\Windows\System32\services.exe'
    condition: selection and not filter
```

### 4.5 Condition Syntax

Supported condition expressions:

| Pattern | Meaning |
|---------|---------|
| `selection` | Single named selection |
| `selection and not filter` | Selection minus exclusion |
| `selection1 or selection2` | Either selection matches |
| `selection1 and selection2` | Both selections match same event |
| `1 of selection*` | Any selection whose name starts with `selection` |
| `all of selection*` | All selections whose name starts with `selection` |
| `not selection` | Negation |

Parentheses and nested expressions are **not supported**. Complex conditions
produce a warning and the rule is skipped.

### 4.6 Not Supported

The following Sigma features are recognized and produce a skip warning:

- `|re` (regex modifier)
- `|base64`, `|base64offset`
- `|cidr`
- Aggregation conditions: `count`, `min`, `max`, `sum`, `avg`, `near`
- Temporal operators
- Parenthesized sub-expressions in conditions
- Field name negation with `not` prefix inside selections

## 5. Data Model

### 5.1 sigma_models.py

```python
@dataclass
class LogSource:
    product: str | None = None    # e.g. "windows"
    service: str | None = None    # e.g. "sysmon", "security"
    category: str | None = None   # e.g. "process_creation"

@dataclass
class SigmaRule:
    title: str
    id: str | None                # Sigma UUID (optional)
    logsource: LogSource
    detection: dict               # raw detection block from YAML
    level: str                    # informational, low, medium, high, critical
    description: str
    tags: list[str]               # e.g. ["attack.execution", "attack.t1059.001"]
    source_path: Path | None      # file this was loaded from
```

### 5.2 sigma_loader.py

```python
def load_sigma_rule(path: Path) -> SigmaRule | None:
    """Parse a single Sigma YAML file. Returns None if unsupported."""

def load_sigma_dir(dir_path: Path) -> list[SigmaRule]:
    """Load all .yml files from a directory. Skips unsupported rules."""
```

### 5.3 sigma_evaluator.py

```python
def evaluate_rule(
    rule: SigmaRule,
    events: list[GeneratedEvent],
) -> list[GeneratedEvent]:
    """Evaluate a Sigma rule against events. Returns matching events."""
```

Internal flow:
1. Filter events by logsource (channel + EID)
2. Parse `detection` block into named selections
3. For each candidate event, evaluate field matchers per selection
4. Apply condition logic across selections
5. Return matching events

## 6. CLI Integration

### 6.1 `artiforge check` Changes

New options:
- `--sigma-dir PATH` — load additional Sigma rules from this directory
- `--sigma-only` — skip built-in rules, only evaluate Sigma rules

### 6.2 Auto-Discovery

When running `artiforge check --lab <id>`, if `artiforge/labs/<id>/sigma/`
exists, its `.yml` files are automatically loaded.

### 6.3 Output Format

```
[ArtiForge] Detection check: Egg-Cellent Resume

  Built-in rules:
  FIRED  DR-001  Suspicious LOLBin Execution      T1218          (2 events)
  NOT    DR-005  Suspicious DNS Query              T1071.004      (0 events)
  ...
  Coverage: 6/13 rules fired (46.2%)

  Sigma rules (3 loaded, 1 skipped):
  FIRED  LOLBin Process Creation                   T1218          (2 events)
  FIRED  Suspicious Service Installation           T1543.003      (1 events)
  NOT    Cloudflared DNS Tunnel                     T1071          (0 events)
  Coverage: 2/3 rules fired (66.7%)

  Skipped (unsupported features):
    advanced_correlation.yml: aggregation condition not supported
```

## 9. Testing Strategy

- **Unit tests for sigma_loader:** Parse valid rule, skip rule with
  unsupported features, handle missing fields
- **Unit tests for sigma_evaluator:** Exact match, wildcard match, contains/
  startswith/endswith modifiers, list OR, `|all` modifier, AND within
  selection, multiple selections with OR/AND conditions, filter exclusion,
  `1 of selection*`, logsource channel filtering
- **Integration test:** Run `artiforge check --lab uc3` with bundled Sigma
  rules, verify expected rules fire
- **CLI test:** `--sigma-dir`, `--sigma-only` flags work correctly

---

## 10. Build Sequence

1. `sigma_models.py` — dataclasses
2. `sigma_loader.py` — YAML parsing + tests
3. `sigma_evaluator.py` — matching engine + tests
4. Starter Sigma rules for UC3
5. CLI integration (`--sigma-dir`, `--sigma-only`, auto-discovery)
6. Integration tests

Steps 1-2 are sequential. Step 3 depends on 1-2. Steps 4-5 depend on 3.

## 11. Out of Scope

- Sigma-to-KQL/SPL conversion
- Regex modifiers (`|re`)
- Aggregation/temporal conditions
- Sigma rule authoring tools
- Trainee rule submission/grading (v0.9)
