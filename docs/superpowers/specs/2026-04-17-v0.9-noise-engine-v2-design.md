# v0.9 Design Spec — Noise Engine v2

> Status: Draft
> Author: d3vn0mi + Claude
> Date: 2026-04-17

---

## 1. Problem Statement

ArtiForge's noise system generates only 3 event types (logon pairs, process
spawns, DNS queries) with uniform temporal distribution. Real enterprise
environments have far richer background activity — file I/O, registry churn,
service lifecycle, network connections, Windows Update traffic — and that
activity clusters around business hours, not uniformly across the day.

Trainees need to filter through realistic noise to find attack signals.
Current noise is too sparse and too evenly distributed to provide that
challenge.

## 2. Solution

Extend the noise engine with 5 new event types, 3 built-in temporal profiles
(office_hours, 24x7_server, developer_workstation), and a preset system that
provides sensible defaults while allowing per-field overrides.

## 3. Design Goals

- Backward compatible — existing labs with `logon_pairs`, `process_spawns`,
  `dns_queries` produce identical output
- Presets set defaults; individual fields override
- Temporal profiles shape when events cluster (not just how many)
- Windows Update noise produces correlated bursts (DNS → connect → download)
- Zero new dependencies

---

## 4. NoiseSpec Model

### 4.1 Updated Fields

```python
class NoiseSpec(BaseModel):
    host: str
    spread_minutes: int = Field(default=60, ge=1)

    # Existing (unchanged)
    logon_pairs: int = Field(default=0, ge=0)
    process_spawns: int = Field(default=0, ge=0)
    dns_queries: int = Field(default=0, ge=0)

    # NEW noise types
    file_operations: int = Field(default=0, ge=0)
    registry_operations: int = Field(default=0, ge=0)
    service_changes: int = Field(default=0, ge=0)
    network_connections: int = Field(default=0, ge=0)
    windows_updates: int = Field(default=0, ge=0)

    # NEW temporal profile
    noise_profile: str | None = None
```

### 4.2 Preset Resolution

When `noise_profile` is set, it provides default counts. Explicitly set
fields (non-zero) override the preset. When `noise_profile` is null, only
explicitly set fields generate noise (current behavior preserved).

### 4.3 Preset Defaults

| Field | `office_hours` | `24x7_server` | `developer_workstation` |
|-------|---------------|---------------|------------------------|
| logon_pairs | 5 | 2 | 3 |
| process_spawns | 10 | 5 | 15 |
| dns_queries | 8 | 4 | 12 |
| file_operations | 5 | 2 | 8 |
| registry_operations | 6 | 3 | 10 |
| service_changes | 2 | 3 | 2 |
| network_connections | 8 | 6 | 10 |
| windows_updates | 2 | 2 | 1 |

### 4.4 YAML Examples

Preset only (all defaults):
```yaml
noise:
  - host: WIN-WS1
    spread_minutes: 120
    noise_profile: office_hours
```

Preset with overrides:
```yaml
noise:
  - host: WIN-WS1
    spread_minutes: 120
    noise_profile: office_hours
    process_spawns: 20        # override preset's 10
    windows_updates: 0        # suppress this type
```

No preset (explicit counts, backward compatible):
```yaml
noise:
  - host: WIN-WS1
    spread_minutes: 60
    logon_pairs: 3
    process_spawns: 5
    dns_queries: 4
    file_operations: 3
    network_connections: 5
```

## 5. Temporal Distribution

### 5.1 Hour-Weighted Sampling

Each profile defines 24 weights (one per hour, 00:00–23:00). Events are
distributed by:

1. Pick an hour using weighted random selection
2. Pick a random minute (0–59) and second (0–59) within that hour
3. Map to an offset within `spread_minutes` from `base_time`

### 5.2 Weight Curves

```python
PROFILES = {
    "office_hours": {
        "weights": [
            1,1,1,1,1,1,1,2,   # 00-07: quiet overnight
            8,10,8,7,           # 08-11: morning ramp-up, peak at 09
            5,9,8,7,            # 12-15: lunch dip, afternoon peak at 13
            6,4,2,1,            # 16-19: end of day taper
            1,1,1,1,            # 20-23: quiet evening
        ],
    },
    "24x7_server": {
        "weights": [
            3,3,2,2,2,2,3,4,   # 00-07: slight overnight dip
            5,5,5,5,            # 08-11: steady
            5,5,5,5,            # 12-15: steady
            5,5,4,4,            # 16-19: steady
            4,3,3,3,            # 20-23: slight taper
        ],
    },
    "developer_workstation": {
        "weights": [
            1,1,1,1,1,1,1,3,   # 00-07: quiet, early start
            9,10,8,6,           # 08-11: deep focus morning
            3,8,9,8,            # 12-15: lunch gap, PM burst
            7,5,3,2,            # 16-19: taper off
            2,3,2,1,            # 20-23: occasional late night
        ],
    },
}
```

### 5.3 Mapping to Spread Window

The temporal profile defines a *shape*, not absolute clock hours. The weights
are used to distribute events proportionally within the `spread_minutes`
window starting at `base_time`. This means:

- `spread_minutes: 60` with `office_hours` → events cluster in the weighted
  portion of that 1-hour window
- `spread_minutes: 480` (8 hours) → events cluster around the
  morning/afternoon peaks within those 8 hours

The mapping divides `spread_minutes` into 24 equal-duration bins. Each bin's
probability is proportional to its weight. To sample a timestamp: pick a bin
via weighted random, then pick a uniform random offset within that bin's time
range. If `spread_minutes` < 24, bins smaller than 1 minute are merged with
their neighbors.

### 5.4 Fallback

When `noise_profile` is null, the engine uses uniform random distribution
(current behavior). This preserves backward compatibility.

## 6. New Noise Generators

### 6.1 `file_operation()` — Sysmon 11 (FileCreate)

Random file creates from benign sources:

**Targets** (weighted random):
- `C:\Users\<user>\AppData\Local\Temp\tmpXXXX.tmp` — temp files
- `C:\Users\<user>\Downloads\document.pdf` — browser downloads
- `C:\Windows\Prefetch\CHROME.EXE-XXXXXXXX.pf` — prefetch
- `C:\Users\<user>\AppData\Local\Microsoft\Office\16.0\OfficeFileCache\*` — Office cache

**Process**: Matched to target — Chrome for downloads, `svchost.exe` for
prefetch, `WINWORD.EXE` for Office cache.

### 6.2 `registry_operation()` — Sysmon 13 (RegistryValueSet)

Random registry writes from benign sources:

**Keys** (weighted random):
- `HKCU\Software\Microsoft\Windows\CurrentVersion\Explorer\UserAssist\*` — shell usage
- `HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Component Based Servicing\*` — CBS
- `HKCU\Software\Microsoft\Office\16.0\Common\*` — Office settings
- `HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\Notifications\*` — notifications

**Process**: `explorer.exe`, `svchost.exe`, `WINWORD.EXE`, `SearchIndexer.exe`

### 6.3 `service_change()` — System 7036 (ServiceStateChange)

Random service state changes:

**Services**: `Spooler`, `Themes`, `WSearch`, `wuauserv`, `BITS`, `Winmgmt`,
`Schedule`, `W32Time`, `EventLog`, `CryptSvc`

**State**: weighted 80% `running` / 20% `stopped`

### 6.4 `network_connection()` — Sysmon 3 (NetworkConnect)

Background HTTPS connections:

**Destinations**: Microsoft CDN (`13.107.x.x`, `20.190.x.x`), Google
(`142.250.x.x`), Akamai (`23.203.x.x`), Cloudflare (`104.16.x.x`)

**Ports**: 443 (90%), 80 (10%)

**Process**: Chrome, Edge, `svchost.exe`, `MsMpEng.exe`

### 6.5 `windows_update()` — Correlated Burst (3 events)

Each call produces a correlated triplet within a 5-second window:

1. **Sysmon 22** (DNS) — query to `update.microsoft.com` or
   `ctldl.windowsupdate.com` or `download.windowsupdate.com`
2. **Sysmon 3** (Network) — HTTPS connection to the resolved IP
3. **Sysmon 11** (FileCreate) — file write to
   `C:\Windows\SoftwareDistribution\Download\<guid>.tmp`

All three share the same process (`svchost.exe`) and use timestamps offset
by 1–3 seconds from each other.

## 8. Testing Strategy

- **Unit tests for noise_profiles:** Preset resolution (defaults, overrides,
  null profile), temporal sampling produces timestamps within spread window,
  weight curves bias toward expected hours
- **Unit tests for new generators:** Each generator produces correct channel,
  EID, required fields, phase_id=0
- **Integration test:** Generate UC3N with `noise_profile: office_hours`,
  verify new event types appear, verify temporal clustering
- **Backward compat test:** Existing UC3N spec produces identical output
  (same event count, same types)

## 9. Build Sequence

1. `noise_profiles.py` — preset defaults, weight curves, `sample_timestamp()`
2. `NoiseSpec` model update — add new fields
3. New generators (file, registry, service, network, windows_update)
4. Update `generate()` — preset resolution + temporal sampling
5. Schema regeneration + tests
6. Integration test with UC3N

Steps 1–2 are sequential. Step 3 can parallelize with 1–2 (generators don't
depend on profiles). Steps 4–6 depend on 1–3.

---

## 10. Out of Scope

- Noise correlation via CorrelationContext (deferred — noise still uses
  independent GUIDs/PIDs)
- Custom user-defined temporal profiles (only 3 built-in presets)
- Per-noise-type temporal profiles (all types in a NoiseSpec share the
  same profile)
- Weekend/weekday distinction (profiles assume a single-day pattern)
