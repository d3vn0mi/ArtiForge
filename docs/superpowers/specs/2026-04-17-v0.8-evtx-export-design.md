# v0.8 Design Spec — Binary EVTX Export (evtxforge)

> Status: Draft
> Author: d3vn0mi + Claude
> Date: 2026-04-17

---

## 1. Problem Statement

ArtiForge produces XML that matches the Windows Event Viewer schema but cannot
be opened by forensic tools (Chainsaw, Hayabusa, EvtxECmd, `Get-WinEvent`) or
Windows Event Viewer itself. These tools require the binary `.evtx` container
format with proper BinXML encoding and checksums.

No Python library exists for *writing* EVTX files. Read-only libraries
(`python-evtx`, `evtx`) are common, but there is no write-side equivalent.

## 2. Solution

Create **evtxforge** — a pure-Python, zero-dependency library for writing
valid `.evtx` files. It lives in `libs/evtxforge/` as a separate package in
the ArtiForge monorepo. ArtiForge consumes it via a thin adapter in
`artiforge/exporters/evtx_exporter.py`.

## 3. Design Goals

- Produce `.evtx` files accepted by: Windows Event Viewer, Chainsaw, Hayabusa,
  EvtxECmd, `Get-WinEvent`, python-evtx (read-back)
- Pure Python, stdlib only (`struct`, `io`, `datetime`, `zlib`)
- Clean public API: `EvtxWriter` context manager, one method per event
- Correct CRC32 checksums, FILETIME encoding, BinXML serialization
- One `.evtx` file per (host, channel) pair in ArtiForge output

## 5. Public API

```python
from evtxforge import EvtxWriter

with EvtxWriter("output.evtx") as writer:
    writer.write_event(
        channel="Security",
        event_id=4624,
        provider_name="Microsoft-Windows-Security-Auditing",
        provider_guid="{54849625-5478-4994-A5BA-3E3B0328C30D}",
        computer="WIN-WS1.lab.local",
        timestamp=datetime(2026, 2, 19, 9, 12, 0, tzinfo=timezone.utc),
        event_data={"TargetUserName": "marcus.webb", "LogonType": "10"},
        record_id=1000,           # optional, auto-increments if omitted
        level=0,                  # 0=info (default)
        task=0,                   # default
        keywords=0x8020000000000000,  # default
    )
```

`EvtxWriter.__exit__` finalizes all chunks (checksums, padding) and writes the
file header.

When `record_id` is omitted, the writer auto-increments from 1. When provided,
the writer uses the given value and updates its internal counter to
`record_id + 1` for subsequent auto-increments.

## 6. EVTX Binary Format

All values are little-endian.

### 6.1 File Header (4096 bytes)

| Offset | Size | Description |
|--------|------|-------------|
| 0 | 8 | Magic: `ElfFile\x00` |
| 8 | 8 | uint64: first chunk number (0) |
| 16 | 8 | uint64: last chunk number |
| 24 | 8 | uint64: next record identifier |
| 32 | 4 | uint32: header size (128) |
| 36 | 2 | uint16: minor version (1) |
| 38 | 2 | uint16: major version (3) |
| 40 | 2 | uint16: header block size (4096) |
| 42 | 2 | uint16: number of chunks |
| 44 | 76 | Reserved (zeroes) |
| 120 | 4 | uint32: flags (0 = clean) |
| 124 | 4 | uint32: CRC32 of bytes [0..119] |
| 128 | 3968 | Padding (zeroes) |

### 6.2 Chunk (65536 bytes)

**Chunk header (512 bytes):**

| Offset | Size | Description |
|--------|------|-------------|
| 0 | 8 | Magic: `ElfChnk\x00` |
| 8 | 8 | uint64: first event record number |
| 16 | 8 | uint64: last event record number |
| 24 | 8 | uint64: first event record ID |
| 32 | 8 | uint64: last event record ID |
| 40 | 4 | uint32: header size (128) |
| 44 | 4 | uint32: last event record data offset (from chunk start) |
| 48 | 4 | uint32: free space offset (from chunk start) |
| 52 | 4 | uint32: event records CRC32 (bytes [512..free_space-1]) |
| 56 | 64 | Reserved (zeroes) |
| 120 | 4 | uint32: flags (0) |
| 124 | 4 | uint32: header CRC32 (bytes [0..119] + [128..511]) |
| 128 | 256 | Common string offset table (zeroed — we don't use caching) |
| 384 | 128 | Template pointer table (zeroed — we don't use templates) |

**Event data area (bytes 512–65535):**
Events packed sequentially. When the next event won't fit, start a new chunk.

### 6.3 Event Record (variable size)

| Offset | Size | Description |
|--------|------|-------------|
| 0 | 4 | Magic: `\x2a\x2a\x00\x00` |
| 4 | 4 | uint32: total record size |
| 8 | 8 | uint64: record identifier |
| 16 | 8 | FILETIME: timestamp (100ns ticks since 1601-01-01 UTC) |
| 24 | var | BinXML body |
| size-4 | 4 | uint32: copy of total record size |

Total size = 24 + len(binxml) + 4.

### 6.4 FILETIME Encoding

```python
_EPOCH_DIFF = 116444736000000000  # 100ns ticks between 1601-01-01 and 1970-01-01

def datetime_to_filetime(dt: datetime) -> int:
    unix_us = int(dt.timestamp() * 10_000_000)
    return unix_us + _EPOCH_DIFF
```

### 6.5 CRC32

Standard CRC-32 (RFC 1952). In Python: `zlib.crc32(data) & 0xFFFFFFFF`.

**File header CRC:** covers bytes [0..119].
**Chunk header CRC:** covers bytes [0..119] + [128..511] (gap at [120..127]).
**Chunk event records CRC:** covers bytes [512..free_space_offset-1].

## 7. BinXML Encoding

Each event becomes a BinXML fragment encoding this XML structure:

```xml
<Event xmlns="http://schemas.microsoft.com/win/2004/08/events/event">
  <System>
    <Provider Name="..." Guid="..."/>
    <EventID>4624</EventID>
    <Version>0</Version>
    <Level>0</Level>
    <Task>0</Task>
    <Keywords>0x8020000000000000</Keywords>
    <TimeCreated SystemTime="2026-02-19T09:12:00.000000000Z"/>
    <EventRecordID>1000</EventRecordID>
    <Channel>Security</Channel>
    <Computer>WIN-WS1.lab.local</Computer>
  </System>
  <EventData>
    <Data Name="TargetUserName">marcus.webb</Data>
    ...
  </EventData>
</Event>
```

### 7.1 Token Values

| Token | Value | Description |
|-------|-------|-------------|
| FragmentHeader | `0x0F 0x01 0x01 0x00` | BinXML fragment start |
| OpenStartElement | `0x01` / `0x41` | Element start (0x41 = has attributes) |
| CloseStartElement | `0x02` | Close `>` |
| CloseEmptyElement | `0x03` | Close `/>` |
| EndElement | `0x04` | End `</tag>` |
| Value | `0x05` / `0x45` | Text value |
| Attribute | `0x06` / `0x46` | Attribute (0x46 = more follow) |
| EOF | `0x00` | End of BinXML |

Bit 6 (0x40) is the "has more data" flag.

### 7.2 Name Structure (BinXmlName)

| Offset | Size | Description |
|--------|------|-------------|
| 0 | 4 | Next offset (0 for simple writer) |
| 4 | 2 | Name hash (can be 0 — not validated on read) |
| 6 | 2 | Character count (excluding NUL) |
| 8 | var | UTF-16LE string + 2-byte NUL terminator |

Total: `4 + 2 + 2 + (chars + 1) * 2`.

### 7.3 String Value

| Offset | Size | Description |
|--------|------|-------------|
| 0 | 1 | Token: `0x05` |
| 1 | 1 | Type: `0x01` (StringType) |
| 2 | 2 | Character count |
| 4 | var | UTF-16LE string (no NUL terminator) |

### 7.4 Encoding Strategy

We use **inline elements** (not templates). This is simpler and produces valid
output — templates are an optimization for log services that write millions of
events with the same schema. For training artifact files with hundreds of
events, inline encoding is correct and sufficient.

The encoder has a single entry point:

```python
def encode_event(
    channel: str,
    event_id: int,
    provider_name: str,
    provider_guid: str,
    computer: str,
    timestamp: datetime,
    event_data: dict[str, str],
    record_id: int,
    level: int = 0,
    task: int = 0,
    keywords: int = 0x8020000000000000,
) -> bytes:
    """Encode a Windows event as a BinXML fragment."""
```

It builds the fragment header, `<Event>` element with `<System>` and
`<EventData>` children, then the EOF token. Returns the complete BinXML
byte sequence ready for embedding in an event record.

## 8. ArtiForge Integration

### 8.1 New Exporter

`artiforge/exporters/evtx_exporter.py`:

```python
def export(bundle: ArtifactBundle, output_dir: Path) -> list[Path]:
    """Write one .evtx file per (host, channel) pair."""
```

Groups `bundle.events` by `(ev.host, ev.channel)`, creates an `EvtxWriter` for
each group, writes events sorted by timestamp.

Output filenames: `WIN-WS1_Security.evtx`, `WIN-WS1_Sysmon.evtx`, etc.

### 8.2 CLI Change

Add `"evtx"` to the `--format` option in `artiforge generate`:

```
artiforge generate --lab uc3 --format evtx
artiforge generate --lab uc3 --format xml,elastic,evtx
```

### 8.3 Dependency

In `artiforge`'s `pyproject.toml` (or `setup.py`):

```toml
dependencies = [
    "evtxforge @ file:libs/evtxforge",
    ...
]
```

## 9. Testing Strategy

### 9.1 evtxforge Unit Tests

- **test_structures.py**: File header packing (magic, version, CRC), chunk
  header packing (magic, CRCs, offsets), event record packing (magic, size,
  FILETIME, trailing size copy)
- **test_binxml.py**: Fragment header encoding, element/attribute/value token
  encoding, name structure encoding, full event XML encoding
- **test_writer.py**: Write single event → verify file size is
  4096 + 65536. Write 100 events → verify multi-chunk. Context manager
  finalizes on exit.

### 9.2 Round-Trip Validation (ArtiForge integration tests)

Use `python-evtx` (test-only dependency) to read back generated `.evtx` files:

- Generate UC3 bundle → export as EVTX → read back with python-evtx
- Verify: event count matches, record IDs match, timestamps match,
  channel names correct, event data fields present
- Verify: files loadable by Chainsaw (if available in CI, otherwise manual)

### 9.3 Manual Validation Targets

Document manual testing steps for:
- Windows Event Viewer (open `.evtx` directly)
- Chainsaw (`chainsaw hunt output/ -s sigma/ --mapping ...`)
- Hayabusa (`hayabusa csv-timeline -d output/`)

## 11. Build Sequence

1. `constants.py` — magic bytes, token values
2. `crc32.py` — checksum wrapper
3. `structures.py` — file header, chunk header, event record packing
4. `binxml.py` — BinXML encoder (fragment header, elements, attributes, values)
5. `writer.py` — EvtxWriter (chunk management, finalization)
6. Unit tests for steps 1-5
7. ArtiForge exporter adapter + CLI integration
8. Round-trip integration tests with python-evtx
9. Manual validation with Chainsaw/Hayabusa (documented steps)

Steps 1-3 are independent. Step 4 depends on 1. Step 5 depends on 2-4.

---

## 12. Out of Scope

- Template-based BinXML encoding (optimization, not needed for correctness)
- Common string caching in chunk headers (optimization)
- Event log service metadata (locale, opcode catalogs)
- Reading/parsing EVTX files (use python-evtx for that)
- Sigma rule integration (separate v0.8 feature, next spec)
- Splunk CIM/HEC export (moved to backlog)
