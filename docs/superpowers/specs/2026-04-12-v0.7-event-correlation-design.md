# v0.7 Design Spec — Event Correlation & Coverage Expansion

> Status: Draft
> Author: d3vn0mi + Claude
> Date: 2026-04-12

---

## 1. Problem Statement

ArtiForge generates forensically unrealistic events. Every identifier —
`LogonId`, `LogonGuid`, `ProcessGuid`, `ProcessId` — is independently
randomised per event. A Sysmon 1 (process create) and Sysmon 5 (process
terminate) for the same process carry different `ProcessGuid` values. A
Security 4624 logon and subsequent 4688 process creation share no `LogonId`.
This prevents trainees from practising the most fundamental forensic skill:
tracing a session across correlated events.

v0.7 introduces three features:
1. **Event Correlation Engine** — shared identifiers flow across related events
2. **Expanded Event Coverage** — 16 new high-value EIDs
3. **Event Sequence Validation** — catches impossible timelines

---

## 2. Design Goals

- Existing labs produce correct output without YAML changes (backward-compat)
- Lab authors don't need to manually wire identifiers for common cases
- Complex scenarios (multiple concurrent sessions/processes) remain expressible
- Validation leverages the correlation engine rather than being a separate system
- Output under `--seed` will change (correlated IDs instead of random) — this is
  intentional and expected

---

## 3. Event Correlation Engine

### 3.1 CorrelationContext

A new class scoped per **(phase, host)** pair. It tracks identifiers that must
be shared across events within the same session/process chain on a single host.

```python
@dataclass
class SessionState:
    """Tracks a single logon session."""
    logon_id: str          # e.g. "0x3a7f1c"
    logon_guid: str        # e.g. "{XXXXXXXX-...}"
    user: str              # username that owns this session

@dataclass
class ProcessState:
    """Tracks a single process lifecycle."""
    process_guid: str
    process_id: str
    image: str             # e.g. "C:\\Windows\\System32\\cmd.exe"
    parent_guid: str | None
    parent_id: str | None
    parent_image: str | None

class CorrelationContext:
    """Per-(phase, host) state shared across event generators."""

    def __init__(self, host: Host):
        self.host = host
        self._sessions: dict[str, SessionState] = {}   # keyed by label or "default"
        self._processes: list[ProcessState] = []        # ordered stack
        self._current_session: SessionState | None = None
        self._current_process: ProcessState | None = None

    # --- Session management ---
    def register_session(self, logon_id: str, logon_guid: str,
                         user: str, label: str = "default") -> SessionState
    def get_session(self, label: str = "default") -> SessionState | None
    @property
    def current_session(self) -> SessionState | None

    # --- Process management ---
    def register_process(self, process_guid: str, process_id: str,
                         image: str, label: str = "default") -> ProcessState
    def get_process(self, label: str = "default") -> ProcessState | None
    @property
    def current_process(self) -> ProcessState | None
```

### 3.2 Field Precedence (Three-Tier)

When a generator resolves a field value, this order applies:

1. **Explicit YAML `fields:` value** — always wins (existing behaviour)
2. **Correlation context value** — new; auto-populated by prior events
3. **Random default** — existing fallback

This means existing labs that specify `fields: { LogonGuid: ... }` keep
working unchanged. Fields not explicitly set now get a correlated value instead
of a random one.

### 3.3 Engine Integration

`dispatch_event()` gains a new `ctx: CorrelationContext` parameter:

```python
def dispatch_event(
    channel: str,
    eid: int,
    fields: dict,
    host: Host,
    user: User | None,
    spec: LabSpec,
    timestamp: datetime,
    ctx: CorrelationContext,       # NEW
) -> dict:
```

Each generator module's `generate()` function passes `ctx` through to
individual EID functions. The engine creates one `CorrelationContext` per
(phase, host) pair before iterating events in that phase:

```python
for phase in phases:
    # One context per host seen in this phase
    phase_contexts: dict[str, CorrelationContext] = {}

    for event_spec in phase.events:
        host = _resolve_host(spec, host_name)
        if host_name not in phase_contexts:
            phase_contexts[host_name] = CorrelationContext(host)
        ctx = phase_contexts[host_name]

        event_data = dispatch_event(..., ctx=ctx)
```

### 3.4 Producer/Consumer Mapping

| **4732** | Consumer | Reads `SubjectLogonId` from current session |

### 3.5 YAML Escape Hatches

For the common case (one session, one process chain per phase/host), no YAML
changes are needed. The engine auto-correlates.

For phases with **multiple concurrent sessions or processes**, lab authors use
optional labels on `EventSpec`:

```yaml
events:
  - channel: Security
    eid: 4624
    session: admin_session       # label — registers this session
    fields:
      TargetUserName: admin
      LogonType: '10'

  - channel: Security
    eid: 4624
    session: svc_session         # second concurrent session
    fields:
      TargetUserName: svc_backup
      LogonType: '3'

  - channel: Sysmon
    eid: 1
    session: admin_session       # binds to admin's LogonGuid
    process: mimikatz            # label for this process
    fields:
      Image: 'C:\Temp\mimikatz.exe'

  - channel: Sysmon
    eid: 10
    process: mimikatz            # reads ProcessGuid from mimikatz
    fields:
      TargetImage: 'C:\Windows\System32\lsass.exe'
```

New optional fields on `EventSpec`:
- `session: str | None = None` — session label (default: `"default"`)
- `process: str | None = None` — process label (default: `"default"`)

### 3.6 Cross-Phase Correlation

**v0.7 scope: correlation resets at phase boundaries.** Each phase starts with
a fresh `CorrelationContext` per host. If a lab needs to carry a session across
phases, the author explicitly provides the identifier via `fields:`.

**Future (v0.8+):** A `continues_session: <phase_id>` directive on a Phase
could auto-carry the session state. This is deferred to keep v0.7 scope tight.

### 3.7 PID Range Expansion

While touching `_pid()`, expand the range to be realistic:

```python
def _pid(category: str = "user") -> str:
    """Generate realistic Windows PID."""
    ranges = {
        "system":  (4, 800),        # System, smss, csrss, wininit
        "service": (800, 5000),     # svchost, lsass, services
        "user":    (5000, 65535),    # user-mode processes
    }
    lo, hi = ranges.get(category, ranges["user"])
    # PIDs are always multiples of 4 on Windows
    return str(random.randint(lo // 4, hi // 4) * 4)
```

Generators will select the appropriate category based on the process image:
- `svchost.exe`, `lsass.exe`, `services.exe` → `"service"`
- `System`, `smss.exe`, `csrss.exe` → `"system"`
- Everything else → `"user"`

### 3.8 Noise Correlation

**Out of scope for v0.7.** The noise generator (`noise.py`) already handles
logon pair correlation (shared `TargetLogonId` within a pair). Extending noise
process spawns to use `CorrelationContext` is deferred to v0.9 (Noise Engine
v2) to keep scope bounded.

## 5. Event Sequence Validation

Extends the existing `artiforge validate --strict` command with correlation-
aware checks. These checks become natural once the correlation engine tracks
identifiers.

### 5.1 New Validation Checks

| Check | Description | Severity |
|-------|-------------|----------|
| **session-before-activity** | Every `SubjectLogonId`-consuming event must be preceded by a 4624 on the same host | warning |
| **process-lifecycle** | Every Sysmon 5 `ProcessGuid` must match a prior Sysmon 1 on the same host | warning |
| **process-before-network** | Sysmon 3 `ProcessGuid` must match a prior Sysmon 1 | warning |
| **parent-exists** | Sysmon 1 `ParentProcessGuid` should match a prior Sysmon 1 (or be the first in chain) | info |
| **no-orphan-logoff** | Every 4634 `TargetLogonId` should match a prior 4624 | warning |
| **temporal-ordering** | Within a phase, events should not precede their logical prerequisites | warning |
| **guid-uniqueness** | No two Sysmon 1 events on the same host should produce the same `ProcessGuid` | error |

### 5.2 Implementation Approach

Validation runs after the full bundle is generated. The engine collects all
produced identifiers during generation (from the `CorrelationContext`) and then
performs a second pass to verify all consumed identifiers were produced.

```
artiforge validate --lab uc3 --strict
```

Output:

```
  Strict checks      WARN  (2 issue(s)):
    ⚠  Phase 3 EID 4688 on WIN-WS2: SubjectLogonId consumed but no prior
       4624 session on this host in this phase
    ⚠  Phase 5 Sysmon 5 on WIN-WS1: ProcessGuid {ABC...} not produced by
       any Sysmon 1 on this host
```

### 5.3 Suppress Mechanism

Lab authors can intentionally create "broken" timelines (e.g., simulating log
gaps or partial captures). A `validation` section on `Phase` allows suppressing
specific checks:

```yaml
- id: 3
  name: "Partial capture — mid-session"
  validation:
    suppress: [session-before-activity]
```

## 7. Backward Compatibility

- **Existing labs (uc3, uc3n):** Will produce **different** output under
  `--seed` because identifiers are now correlated instead of random. The events
  are *more correct*, not broken. No YAML changes required.
- **Generator API:** `dispatch_event()` gains a required `ctx` parameter. Any
  external code calling this function directly must be updated. Internal callers
  (engine.py, noise.py) are updated in this release.
- **Noise generator:** Receives a `ctx=None` sentinel and continues using its
  own internal correlation for logon pairs. Full noise correlation is deferred
  to v0.9.
- **Lab schema version:** Bumped to `"2"`. Engine accepts both `"1"` and `"2"`
  with a deprecation warning for `"1"`.

## 8. Out of Scope

- Cross-phase correlation (explicit `continues_session` directive) — deferred to v0.8+
- Noise generator correlation via CorrelationContext — deferred to v0.9
- EVTX binary export — v0.8
- Sigma rule integration — v0.8
- Real file hashes from public corpus — future consideration

## 9. Build Sequence

Implementation should proceed in this order:

1. **CorrelationContext class** (`correlation.py`) + unit tests
2. **Engine integration** — create contexts, pass to dispatch, update dispatch signature
3. **Security generators** — retrofit existing EIDs to use context (producers first: 4624, then consumers)
4. **Sysmon generators** — retrofit existing EIDs (Sysmon 1 as producer, then consumers)
5. **PID range expansion** — update `_pid()` in both modules
6. **New Security EIDs** (1102, 4697, 4703, 4719, 4735)
7. **New Sysmon EIDs** (6, 15, 16, 24, 26)
8. **New System EIDs** (7031, 7034)
9. **New PowerShell EIDs** (4105, 4106, 40961, 40962)
10. **EventSpec model update** — add `session`, `process`, `validation` fields
11. **Validation checks** — extend `--strict` with correlation-aware checks
12. **Schema regeneration** + existing test updates
13. **Integration test** — run uc3 with correlation, verify GUID/LogonId consistency

Steps 6–9 are independent and can be parallelised.
