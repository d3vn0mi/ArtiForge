# v0.9 Design Spec — Linux Auditd Channel

> Status: Draft
> Author: d3vn0mi + Claude
> Date: 2026-04-17

---

## 1. Problem Statement

ArtiForge is Windows-only. Linux is the dominant server OS, and `auditd` is
the primary audit framework for Linux incident response. Training labs that
cover lateral movement from Windows to Linux, Linux-only server compromises,
or mixed environments are currently impossible.

## 2. Solution

Add a `linux_auditd` generator module with 7 record types, a raw `audit.log`
exporter, ECS-mapped NDJSON output (matching Auditbeat's schema), and a
platform-aware host model so labs can mix Windows and Linux hosts.

## 3. Design Goals

- Labs can mix Windows and Linux hosts in the same infrastructure
- Backward compatible — existing Windows-only labs unchanged
- Raw `audit.log` output for forensic CLI tools (`ausearch`, `aureport`)
- ECS NDJSON output for Kibana/Elasticsearch (Auditbeat-compatible)
- Auditd events linked by shared `audit_id` (timestamp:serial)
- CorrelationContext works for Linux events (same session/process tracking)

## 4. Host Platform Model

### 4.1 New `platform` Field

```python
class Host(BaseModel):
    name: str
    ip: str
    fqdn: str
    os: str = "Windows 10"
    platform: str = "windows"     # NEW — "windows" | "linux"
    sid_prefix: str = "S-1-5-21-1111111111-2222222222-3333333333"
    users: list[User] = Field(default_factory=list)
```

Default is `"windows"` for full backward compatibility.

### 4.2 YAML Usage

```yaml
infrastructure:
  domain: lab.local
  hosts:
    WIN-WS1:
      ip: 10.10.10.10
      fqdn: WIN-WS1.lab.local
      platform: windows
      os: "Windows 10 22H2"
      sid_prefix: "S-1-5-21-..."
      users:
        - username: marcus.webb
          domain: LAB
          rid: 1001

    LNX-WEB1:
      ip: 10.10.10.50
      fqdn: lnx-web1.lab.local
      platform: linux
      os: "Ubuntu 22.04 LTS"
      users:
        - username: www-data
          domain: lnx-web1
          rid: 33
        - username: root
          domain: lnx-web1
          rid: 0
```

For Linux hosts, `sid_prefix` is ignored. `User.rid` maps to Linux UID.
`User.domain` is the hostname (Linux convention).

### 4.3 Engine Routing

The engine uses `host.platform` to select the channel map:

- `platform: "windows"` → existing channels (Security, Sysmon, System, etc.)
- `platform: "linux"` → `Auditd` channel only

If a lab author specifies `channel: Security` on a Linux host, the engine
raises a validation error.

## 5. Auditd Record Types

### 5.1 Record Type → EID Mapping

Each auditd record type gets a numeric EID for internal tracking:

| Record Type | EID | Description |
|-------------|-----|-------------|
| SYSCALL | 1300 | System call metadata |
| EXECVE | 1309 | Command execution arguments |
| PATH | 1302 | File path accessed |
| SOCKADDR | 1306 | Network socket address |
| USER_AUTH | 1100 | PAM authentication |
| USER_LOGIN | 1101 | Login session |
| CRED_ACQ | 1103 | Credential acquisition (su/sudo) |

### 5.2 Audit ID Linking

Related auditd records share an `audit_id` composed of `timestamp:serial`.
For example, a process execution produces a linked set:

```
type=SYSCALL msg=audit(1771398720.123:1001): ...
type=EXECVE msg=audit(1771398720.123:1001): ...
type=PATH msg=audit(1771398720.123:1001): ...
```

The generator tracks the current serial number per (phase, host) via the
existing `CorrelationContext`. A single "event" in the lab YAML can produce
multiple linked records (e.g., SYSCALL + EXECVE + PATH for a process exec).

## 6. Lab YAML — Linux Events

Linux events use `channel: Auditd` with the record type as a string EID
alias or numeric EID:

```yaml
phases:
  - id: 1
    name: "SSH Brute Force"
    host: LNX-WEB1
    user: root
    events:
      - channel: Auditd
        eid: USER_AUTH             # string alias
        offset_seconds: 0
        fields:
          acct: root
          exe: /usr/sbin/sshd
          hostname: 10.10.10.10
          res: failed
        repeat: 5
        repeat_gap_seconds: 2

      - channel: Auditd
        eid: USER_LOGIN
        offset_seconds: 12
        fields:
          acct: root
          exe: /usr/sbin/sshd
          hostname: 10.10.10.10
          res: success
```

The generator resolves string aliases (`USER_AUTH` → 1100) internally.

### 6.1 Compound Events

Some actions produce multiple linked records. For convenience, the generator
supports compound EID aliases:

| Alias | Records produced |
|-------|-----------------|
| `EXEC` | SYSCALL(59) + EXECVE + PATH |
| `CONNECT` | SYSCALL(42) + SOCKADDR |

```yaml
      - channel: Auditd
        eid: EXEC                  # compound: produces SYSCALL + EXECVE + PATH
        offset_seconds: 15
        fields:
          exe: /usr/bin/bash
          args: ["-c", "id && uname -a"]
```

Compound events share the same `audit_id` and are emitted as sequential
`GeneratedEvent` objects.

## 7. Exporters

### 7.1 Raw audit.log Exporter

`artiforge/exporters/auditd_exporter.py`

Produces one `audit.log` per Linux host. Format:

```
type=SYSCALL msg=audit(1771398720.123:1001): arch=c000003e syscall=59 success=yes exit=0 ppid=1 pid=5678 auid=1000 uid=0 gid=0 euid=0 suid=0 fsuid=0 egid=0 sgid=0 fsgid=0 tty=pts0 ses=1 comm="bash" exe="/usr/bin/bash" key="exec_monitor"
type=EXECVE msg=audit(1771398720.123:1001): argc=3 a0="bash" a1="-c" a2="id && uname -a"
type=PATH msg=audit(1771398720.123:1001): item=0 name="/usr/bin/bash" inode=131073 dev=08:01 mode=0100755 ouid=0 ogid=0 rdev=00:00 nametype=NORMAL
```

Timestamp in the `msg=audit(...)` field uses Unix epoch seconds with
millisecond precision.

```python
def export(bundle: ArtifactBundle, output_dir: Path) -> list[Path]:
    """Write one audit.log per Linux host. Return written paths."""
```

### 7.2 ECS NDJSON Update

`artiforge/exporters/elastic.py` gains Linux-aware field mappings:

- `event.module: "auditd"` for Auditd channel events
- `event.category` mapped per record type (authentication, process, file, network)
- `process.name`, `process.executable`, `process.pid`, `process.args`
- `user.name`, `user.id`
- `source.ip`, `source.port`, `destination.ip`, `destination.port`
- `auditd.log.record_type`, `auditd.log.sequence`

### 7.3 CLI

Add `"auditd"` to `--format`:

```bash
artiforge generate --lab uc5 --format auditd,elastic
artiforge generate --lab uc3 --format xml,elastic,evtx     # still works
```

The `auditd` format is only applied to Linux hosts. Windows hosts in the
same bundle are silently skipped by the auditd exporter (and vice versa —
the XML/EVTX exporters skip Linux hosts).

## 9. Testing Strategy

- **Generator unit tests:** Each record type produces correct fields, string
  alias resolution, compound event linking (shared audit_id)
- **Exporter tests:** Raw audit.log format matches expected key=value output,
  one file per Linux host, Windows hosts skipped
- **ECS mapping tests:** Linux events produce correct Auditbeat-compatible
  ECS fields
- **Platform routing tests:** Linux host rejects Windows channels, Windows
  host rejects Auditd channel
- **Mixed lab test:** Lab with both Windows and Linux hosts generates correct
  events for each platform
- **Backward compat:** Existing labs (uc3, uc3n) unchanged — `platform`
  defaults to `"windows"`

## 10. Build Sequence

1. `Host.platform` model update + backward compat tests
2. `linux_auditd.py` — record type generators (SYSCALL, EXECVE, PATH, etc.)
3. Compound event aliases (EXEC, CONNECT)
4. `auditd_exporter.py` — raw audit.log output
5. `elastic.py` — Linux ECS mappings
6. `__init__.py` + `cli.py` — platform-aware routing + CLI format
7. Schema regen + integration tests

Steps 2-3 are sequential. Steps 4-5 depend on 2. Step 6 depends on 2-5.
